# Android App Development Workflow

## Overview

A battle-tested pipeline for building Android apps with Flutter/Dart. Uses a 3-agent model: main agent (orchestrator), developer agents (implementers), tester agent (verifier). Based on real project experience. Covers every phase from requirement discussion to emulator testing.

**Key philosophy**: Each phase produces a concrete deliverable. Phase output feeds into the next phase. Never start coding before PRD is written.

## Phase 0: Requirements & Design

**Deliverable**: `docs/PRD.md` + `docs/DEVELOPMENT.md` + `docs/PROJECT.md` + `CLAUDE.md`

### Step 0.1 — Requirement Discussion

Use `AskUserQuestion` to establish these decision points. Ask them one at a time, waiting for user confirmation before proceeding.

```
Q1: Product positioning?
   A. Entertainment / casual tool (disclaimers required)
   B. Serious / professional tool
   C. Educational / learning aid

Q2: Core feature modules?
   → List each as a separate module. Modules must be independent (no cross-UI coupling).

Q3: Target platform?
   A. Android only (flutter build apk)
   B. Android + iOS
   C. Cross-platform (add macOS/Windows)

Q4: Tech stack?
   - State management: Provider / Riverpod / Bloc
   - Local DB: sqflite / drift / hive
   - LLM API: Claude / DeepSeek / OpenAI
   - Offline strategy: degrade gracefully vs prompt "needs network"
   - UI style: Material Design 3 vs custom themed

Q5: Development approach?
   A. Linear (one phase at a time, modular)
   B. Multi-agent parallel (main agent orchestrates dev + tester agents)
```

After user answers, confirm understanding in a short summary before proceeding to Step 0.2.

### Step 0.2 — Write PRD

Call the **`prd`** skill (or `prd-skill`):
```
Skill: prd
Args: 撰写关于 [app name] 的完整 PRD，包含：
- 产品定位（从 Step 0.1 获取的信息）
- [N] 大功能模块
- 目标用户画像
- P0/P1/P2 功能优先级
- 用户故事 + 验收标准
- 交互流程图（ASCII 或文字描述）
- 页面设计规格
- 数据模型概要（字段级）
- 非功能性需求（性能/兼容性/安全）
- 验收标准
```

Save output to `docs/PRD.md`.

### Step 0.3 — Write Development Documents

Create three files manually (do NOT delegate these):

**`docs/DEVELOPMENT.md`** — Include:
- Project overview table (name, platform, language, framework)
- Directory structure (full tree for `lib/`)
- Architecture layers (UI → Services → Engines → Models → Data)
- Core algorithm coverage table
- Library dependencies with versions
- Phase-by-phase development plan (Phase 1/2/3/4/5)
- Key architecture diagrams (data flow for each module)

**`docs/PROJECT.md`** — Include:
- Project info table
- Phase tracking table with status (✅⬜) and completion dates
- Design decision log
- Reference project links

**`CLAUDE.md`** — Use the **`andrej-karpathy-skills:karpathy-guidelines`** skill for guidance. Include:
- Project overview and tech stack (compact)
- Karpathy's 4 rules (think before code, simplicity, surgical changes, goal-driven)
- 3-agent collaboration rules (with flow diagram)
- Modular development principles
- Phase delivery list
- Key design decisions (short)
- Reference data sources

### Step 0.4 — Create Tasks & Memory

Create tasks for each phase using `TaskCreate`:
```
TaskCreate:
  subject: "P1: Data Layer + Algorithm Engines"
  description: "Models, constants, engines, database"
```

Save key design decisions to project memory for continuity across phases.

### Documents Checklist

Run through this checklist before proceeding to Phase 1:
- [ ] `docs/PRD.md` — generated by prd skill
- [ ] `docs/DEVELOPMENT.md` — architecture, structure, algorithms, phases
- [ ] `docs/PROJECT.md` — progress tracking, milestones, decisions
- [ ] `CLAUDE.md` — AI guidelines, Karpathy rules, 3-agent model

## Phase 1: Data Layer + Algorithm Engines

**Deliverable**: All `lib/models/*`, `lib/data/constants/*`, `lib/data/database/*`, `lib/engines/*`. Must pass `dart analyze lib/` with zero errors.

### Step 1.1 — Flutter Project Setup

```bash
flutter create --org com.yourcompany --project-name your_app .
```

Then configure `pubspec.yaml`:
- Add all required dependencies
- Add asset paths for JSON data files
- Run `flutter pub get`

Create directory structure:
```
lib/
├── models/<module>/
├── data/constants/  database/
├── engines/
├── services/
├── pages/<module>/
├── widgets/
└── app/
```

### Step 1.2 — Write Models + Constants + Engines

**Order of execution (sequential dependency)**:
1. Models first (no dependencies on other code)
2. Constants & data files (depend on models for type references)
3. Database layer (depends on models)
4. Algorithm engines (depend on models + constants)

**Parallelization strategy**: Within each step, spawn developer agents for INDEPENDENT modules simultaneously:
```
Same turn → Agent 1: Write meihua model + engine
             Agent 2: Write tarot model + engine  
             Agent 3: Write tarot_cards.json (large data file)
```

**Agent dispatch prompt template**:
```
Agent:
  description: "[module] implementation"
  prompt: |
    Write all [module] files for the Flutter project at [path].
    
    Context from PRD:
    - [data model spec]
    - [algorithm spec]
    
    Existing dependencies you can import:
    - [list shared models/constants]
    
    Requirements:
    - Follow the existing code style
    - All static analysis must pass
    - Write only the files listed below:
      [file list]
    
    Save each file to: lib/[path]
```

### Step 1.3 — Verify

```bash
dart analyze lib/
# Must show: No issues found!
```

## Phase 2: Standalone Module UI

**Deliverable**: App theme + common widgets + each module's UI pages + placeholder LLM readings.

### Step 2.1 — App Foundation (write directly, don't delegate)

- `lib/app/theme.dart`: Material Design 3 `ColorScheme.fromSeed`, define module-specific colors
- `lib/app/app.dart`: `MaterialApp` + `onGenerateRoute` handler with all routes registered upfront
- `lib/main.dart`: Entry point with async initialization

### Step 2.2 — Common Widgets (write directly)

Components to build in `lib/widgets/`:
- Animation widgets (card flip with `AnimationController` + `Transform.rotateY`)
- Data visualization (line-based hexagram view, comparison view)
- Loading overlay / error display
- Disclaimer text component

### Step 2.3 — Module UI Pages (delegate to agents)

Spawn developer agents in parallel for EACH module:
```
Same turn → Agent A: Write all meihua pages
             Agent B: Write all tarot pages  
```

Each agent gets:
- The module's data models as context
- The engine's API (function signatures)
- The LLM service abstract interface
- The app's theme colors to use
- Route names to register

### Step 2.4 — LLM Service Abstraction (write directly)

```dart
abstract class LlmService {
  static LlmService instance = _PlaceholderService();
  static void configure(LlmService service) => instance = service;
  Future<String> getReading(YourResult result);
}
```

Placeholder returns fixed text for UI testing.

**Deliverable**: Mapper logic + fusion UI chain + real LLM API implementation.

### Step 3.1 — Mapping Logic (write directly)

Build a strategy pattern: module A's output → parameters for module B's engine.

```dart
class FusionMapper {
  static Map<String, dynamic> getStrategy(String relation) { ... }
}
```

### Step 3.2 — Fusion UI (delegate to agent)

Single agent for 4 connected pages: input → intermediate → reveal → reading.

### Step 3.3 — LLM API Service (write directly)

Implement the real service with:
- System prompts: one per reading type
- Structured user messages: serialize all module data
- Error handling: timeout / network failure → user-friendly text
- Configuration via encrypted stored API key

**Deliverable**: History UI + settings + full integration test.

### Step 5.1 — History Module
- List page with type filter chips + swipe-to-delete + empty state
- Detail page with JSON tree view
- Auto-save helper: hook into each result page's future completion

### Step 5.2 — Settings Page
```dart

flutter_secure_storage → saveApiKey / readApiKey / deleteApiKey

// Connection test
DeepseekLlmService → call API with placeholder data → show success/failure
```

### Step 5.3 — Activation in main.dart
```dart
final hasKey = await SecureStorage.hasApiKey();
if (hasKey) LlmService.configure(DeepseekLlmService());
```

### Step 5.4 — Android Build Config
- Set `minSdk = 26` in `android/app/build.gradle.kts`
- Accept SDK licenses: `yes | sdkmanager --licenses`

### Step 5.5 — Emulator Testing Flow
```bash
# 1. Create emulator
avdmanager create avd -n test -k "system-images;android-34;google_apis_playstore;arm64-v8a" --force

# 2. Start emulator
export ANDROID_HOME=/path/to/sdk
$ANDROID_HOME/emulator/emulator -avd test -no-audio -no-window &

# 3. Wait for boot
adb wait-for-device
sleep 30  # wait for boot_completed

# 4. Build and run
flutter build apk --debug
flutter install

# 5. Navigate and test each module via adb
adb -s emulator-5554 shell input tap x y   # tap to navigate
adb -s emulator-5554 shell input text "3"  # input text
adb -s emulator-5554 exec-out screencap -p > screenshot.png

# 6. Check for errors
adb -s emulator-5554 logcat -d | grep -iE "flutter" | grep -vE "Choreographer"
```

## 3-Agent Orchestration Protocol

### Main Agent (you, the current session)
- **Owns**: Phase transitions, task splitting, code review, architectural decisions
- **DO**: Write foundation files directly (theme, routing, main, LLM abstraction, mapping logic)
- **DELEGATE**: Large data files (>200 lines), complete module UI pages, algorithm engines
- **VERIFY**: Always run `dart analyze` after agent completes before merging
- **CONTEXT MANAGEMENT**: When conversation approaches 40% context limit, tell the user and suggest `/compact` before continuing

### Developer Agent (spawned via Agent tool)
- **Receives**: Self-contained prompt with all context needed (does NOT share your conversation history)
- **Owns**: Writing code + self-testing (`dart analyze` must pass)
- **Prompt pattern**:
  ```
  Agent:
    description: "[specific task]"
    subagent_type: general-purpose
    prompt: |
      [Project context — what this project is]
      [Input spec — what data/API to use]
      [File list — exactly which files to create]
      [Constraints — coding standards, imports, style]
      Only create files, do NOT run commands.
  ```

### Tester Agent (spawned after a module is complete)
- **Receives**: The module's PRD acceptance criteria + code paths
- **Owns**: Writing test cases, executing `dart analyze`, running `flutter test`, reporting bugs
- **Output format**:
  ```
  ## Test Report
  ### Static Analysis: pass/fail
  ### Module: [name]
  - [验收项]: ✅/❌ + reason
  ### Bugs Found:
  1. Severity: Critical/High/Medium/Low — Description — File:line
  ```

### Agent Dispatch Timing

```
Phase start → Main writes foundation → Spawn developers (parallel) → Review → Merge
                 ↓
         Module done → Spawn tester → Review bug report → Fix → Regression
                 ↓
         All modules done → Next phase
```

## Context Management

Flutter/Android projects generate significant code. To avoid context overflow:

1. **Use TaskCreate to track all subtasks** — tasks survive context compression
2. **Delegate aggressively** — large data files (tarot_cards.json, 64 hexagrams) always go to subagents
3. **Save memory** — write key decisions to project memory directory after each phase
4. **/compact** — when context > 40%, suggest the user run `/compact` and summarize current state in CLAUDE.md
5. **Agent isolation** — each developer agent works in isolation, their context doesn't affect yours

## Common Pitfalls Reference

### Algorithm: Trigram number mapping
3-bit binary pattern (0-7) ≠ 先天八卦数 (1-8). Correct mapping:
```
Pattern→Value: 111=7→1(乾), 110=6→2(兑), 101=5→3(离), 100=4→4(震)
               011=3→5(巽), 010=2→6(坎), 001=1→7(艮), 000=0→8(坤)
Formula: value == 0 ? 8 : 8 - value
```

### Algorithm: Hexagram line order convention
MSB/LSB convention must be consistent between data encoding and reading. Verify with 2-3 known hexagrams. The convention used in the reference project is: MSB=bottom line (初爻), LSB=top line (上爻).

### Algorithm: Hu/Bian Gua bit extraction
When extracting 3 lines for interaction/change trigrams from a 6-line hexagram, use `bit2=bottom, bit1=middle, bit0=top` consistently across methods.

### Algorithm: BaZi year pillar
Chinese astrology year starts at Lichun (Feb 4±1), NOT lunar New Year. Birth dates before Lichun use previous year's stem-branch.

### Flutter: Dart 3.12+ compatibility
- Use `withValues(alpha:)` NOT `withOpacity()` (deprecated)
- Route arguments: received via `ModalRoute.of(context)!.settings.arguments`
- JSON assets must be declared in `pubspec.yaml` `flutter: assets:`
- `DropdownButtonFormField`: `value` is deprecated, use `initialValue` in Flutter 3.33+

### Flutter: Navigation pattern
```
// Sending page:
Navigator.pushNamed(context, '/route', arguments: yourObject);

// Receiving page (in initState or didChangeDependencies):
final data = ModalRoute.of(context)!.settings.arguments as YourType;
```

### Multi-Agent: Prompt design
- Each agent prompt must be self-contained (the agent has NO context from your conversation)
- Include: project path, importable existing code, file list, coding standards
- For data files: specify exact JSON/dart format
- Use `run_in_background: true` for long-running agents (>30s expected)
